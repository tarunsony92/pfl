# Milestone 4: Next.js 14 Frontend — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver the complete Next.js 14 App Router frontend that makes the full M1–M3 pipeline operable through a browser. Ships all 10 screens (login, case list, upload wizard, case detail with 6 tabs, reupload, admin dedupe snapshots, admin user management, profile/settings). Also closes backend security follow-ups M1-C1 (HttpOnly refresh cookie) and M1-I2 (user deactivation endpoint) and hardens the `list_cases_endpoint` against underwriter over-fetching.

**Architecture:** Next.js 14 App Router with RSC page shells + Client Component islands. All API calls are proxied through `/api/proxy/[...path]` Route Handlers (server-side cookie forwarding, token-in-memory refresh). SWR for client-side data fetching with adaptive polling. No direct browser-to-backend calls. Single docker-compose service `pfl-web` on port 3000.

**Tech Stack:** Next.js 14 App Router · TypeScript (strict) · Tailwind CSS · shadcn/ui · react-hook-form + zod · SWR · Playwright (E2E) · Vitest (unit/component). Node 20 Alpine Docker image.

**Builds on:** M3 (tagged `m3-ingestion-workers`).

**Spec reference:** `docs/superpowers/specs/2026-04-18-m4-frontend-design.md` — implementers should read the relevant section when prompted.

**Definition of done:** Spec §17. All screens functional end-to-end against docker-compose stack; Playwright E2E green; Vitest ≥ 80% coverage; Lighthouse ≥ 90 on `/login` and `/cases`; ruff + mypy clean on backend patches; tag `m4-frontend`.

## Task 11: Route Handler proxy + middleware

**Files:** `frontend/src/app/api/proxy/[...path]/route.ts`, `frontend/src/middleware.ts`

**Reference:** Spec §4.3, §5.5, §8.1, §8.2

- [ ] `src/app/api/proxy/[...path]/route.ts`:
  - Export handlers for GET, POST, PATCH, DELETE, PUT.
  - Each handler: read path segments from Next.js `params.path`; reconstruct backend URL as `${BACKEND_INTERNAL_URL}/${path.join('/')}?${queryString}`.
  - Attach `Authorization: Bearer {token}` from `server-auth.ts`.
  - Stream request body through for POST/PATCH/PUT (handle multipart).
  - On 401 from backend: attempt single-flight refresh via `server-auth.ts` mutex; retry original request; on second 401 clear token and return 401 to client.
  - Forward `Set-Cookie` headers from backend responses to client (critical for `refresh_token` cookie and logout cookie clearing).
  - On login success (`POST auth/login` 200): extract access token from response body; call `storeAccessToken`; set `pfl_auth=1` non-HttpOnly cookie.
- [ ] `src/middleware.ts`:
  - Runs on all paths except `/_next/*`, `/favicon.ico`, `/api/health`.
  - If `pfl_auth` cookie absent and path is not `/login`: redirect to `/login?next={pathname}`.
  - If `pfl_auth` present: continue.
  - Seed `csrf_token` cookie if absent: generate 32-byte hex; set as non-HttpOnly `SameSite=Strict` cookie.
