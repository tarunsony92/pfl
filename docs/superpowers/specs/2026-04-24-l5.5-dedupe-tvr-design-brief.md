# L5.5 · Dedupe + TVR verification — design brief

**Date:** 2026-04-24
**Status:** Design brief (stub) — full spec + plan to be written when this moves into an active sprint. Lands in the Part B / post-Part-A backlog.
**Owner:** saksham@pflfinance.com

> This is a brief, not an implementation-ready spec. It captures the
> user's ask, the sample-data findings, and my recommendation on
> phasing + technical approach so the full spec can be fleshed out
> quickly when we're ready to build.

## 1. The ask

Add a new pipeline level **L5.5 · Dedupe + TVR** between L5 (Scoring) and
L6 (Decisioning). The new level:

1. Expects two new artifact types auto-extracted from the case ZIP:
   - **Dedupe report** (xlsx) — identity-dedupe output against the
     Finpage customer store. Sample columns: Customer Id, Type, Full
     Name, Aadhaar Id, Voter Card, Pan Card, Nrega, Passport,
     Driving License, DOB, Father Name, Mother Name, Bank Account,
     IFSC, UID Enc, Mobile No.
   - **TVR audio** (mp3) — Tele-Verification Report the assessor
     conducts with the applicant, typically in Hindi.
2. **Dedupe logic:**
   - Blank sheet (only headers) → auto-PASS ("no duplicate on file").
   - Any data rows → flag for assessor review + MD approval before
     turning green.
3. **TVR logic:**
   - Presence check: file is ingested ⇒ level can proceed.
   - Absent → BLOCKED until uploaded.
   - (Optional Phase 2) Content analysis: transcribe + cross-check
     the applicant's statements against L1–L5 extracted fields +
     suggest NBFC-standard follow-up questions.
4. Must gate L6 Decisioning — L6 does not run until L5.5 is
   PASSED or PASSED_WITH_MD_OVERRIDE.

## 2. Sample-data findings (from files user attached)

- `Customer_Dedupe (1).xlsx` — single sheet, 2 rows × 16 cols. Row 1
  is a "Finpage - Home" title row; row 2 is the column headers. No
  data rows. This is the **empty-dedupe** case → auto-PASS per §1.2.
- `f4dcb43843ca57f7870ca0e4f06d1a4n.mp3` — 377 KB, MPEG ADTS v2.5
  layer III, 8 kHz mono, 8 kbps. Standard telephony-quality Hindi
  audio; ~6 min duration. The filename looks like a Finpage content
  hash (no customer name), so we cannot rely on filename pattern
  matching for ingestion categorisation — need MIME / extension
  matching.

## 3. Phasing recommendation

**Ship in two phases.** Phase 1 is the full level without audio
content analysis; Phase 2 adds audio understanding.

### Phase 1 — presence-check only · high confidence, low cost

- **Ingestion** — extend the ZIP categoriser to recognise:
  - `*.xlsx` with a filename containing "dedupe" OR "duplicate" →
    `DEDUPE_REPORT`.
  - `*.mp3` / `*.wav` / `*.m4a` → `TVR_AUDIO_RECORDING` (only one per
    case expected; if multiple, keep the largest).
- **Dedupe extractor** — parse the xlsx, count data rows (rows
  after the header row that contain a non-empty Customer Id). Emit
  `{row_count, matched_fields: [list of columns with non-null data]}`
  on the CaseExtraction row.
- **L5.5 orchestrator** — runs two cross-checks:
  - `dedupe_clear`: blank → PASS. Non-blank → WARNING with the
    matched-rows payload as evidence; requires assessor resolve.
  - `tvr_present`: TVR audio artifact exists on case → PASS.
    Missing → CRITICAL.
- **Gate update** — add L5.5 to the gate-check expression in
  `gate_open_for_phase_1` so L6 only runs after L5.5 passes.
- **Frontend** — add L5.5 to the `LEVELS` array + `RULE_CATALOG`
  with the two rules above; the existing level-card infrastructure
  handles rendering. Surface the dedupe rows (if any) as a simple
  table in the L5.5 detail panel.

**Estimated effort:** 1.5–2 days. No external AI call, deterministic,
testable end-to-end.

### Phase 2 — TVR audio understanding · opt-in, higher cost

Only ship when Phase 1 is stable and the business wants the extra
insight.

- **Transcription** — Claude does not currently take raw audio
  inputs on the Messages API. Two paths:
  1. OpenAI Whisper (`whisper-1` model) — ~$0.006 / minute. ~$0.04
     for a 6-min TVR. Hindi supported natively, returns plain text
     or English-translated text.
  2. Google Cloud Speech-to-Text — similar cost, bigger vendor
     sprawl. Whisper preferred unless there's a policy against
     OpenAI.
- **Cross-check** — feed the Hindi transcript + the English
  translation + a structured L1–L5 snapshot to Opus
  (`TvrCrossChecker` service, new). Opus returns:
  - `statements_aligned` — per-field verdict: applicant's stated
    address vs L1 Aadhaar/GPS address; stated business type vs L3
    scorer output; stated income vs L2 declared income; etc.
  - `follow_up_questions` — NBFC-industry-standard clarifications
    the assessor should ask in a second call if any signal is
    unclear. For example: "CIBIL report shows a loan from Bank X
    that the applicant did not mention — confirm end use."
  - `risk_notes` — anything that looks rehearsed, scripted, or
    evasive.
- **Issues emitted** — each `statements_aligned` item whose
  verdict is "mismatch" becomes a WARNING; `risk_notes` entries
  become concern cards.
- **Cost ceiling** — per-case budget ~$0.05 (Whisper) + ~$0.05
  (Opus narrative) = $0.10 / case. Gate behind a feature flag so
  the finance team can disable if QPS spikes.
- **Low-confidence path** — if the transcript is < 100 words OR
  Whisper's language detection confidence is < 0.6, do not run the
  cross-check; emit a WARNING that the audio was too short /
  unclear for analysis. Prevents garbage-in, garbage-out at the
  Opus layer.

**Estimated effort:** 2.5–3 days.

## 4. Dependencies / risks / open questions

- **Gate-check change touches L6.** The Ajay case currently has L6
  `COMPLETED`; adding L5.5 to the gate means Ajay's gate may flip
  shut until L5.5 passes. Plan for a one-time re-run after deploy.
- **ZIP ingestion is shared with every case.** Adding two new
  subtypes doesn't risk regressions to existing cases (categoriser
  only adds a branch), but a new XLSX extractor for dedupe needs to
  coexist with the Excel-based CAM / cibil extractors — confirm no
  filename collision.
- **Assessor → MD escalation for non-blank dedupe** needs the same
  MD-override flow L1 has. Reuse existing issue-status state
  machine.
- **Hindi transcript storage.** If Phase 2 ships, the transcript is
  now sensitive PII sitting on our storage — confirm retention /
  redaction policy with the assessor team before writing.
- **Multiple dedupe xlsx files in a single ZIP** — pick the newest
  by metadata timestamp.
- **Whisper vs on-prem.** On-prem whisper.cpp is free but the ops
  cost (GPU / Apple Silicon box) dwarfs the Whisper API fee at our
  volume. Use the API for now.

## 5. Out of scope (do not add to Phase 1)

- Biometric voice-match against a prior call recording.
- Sentiment analysis / stress detection on the audio waveform.
- Auto-approving the level without assessor touching it when a
  non-blank dedupe fires (policy-sensitive).
- Per-field dedupe scoring (fuzzy match on name, DOB). If the
  Finpage report already has a confidence column we'll use it
  verbatim; designing our own dedupe scorer is a separate product
  ask.
